# Conversation Flow for Submission Log

이 파일은 실제 제출 로그를 만들 때 사용할 대화 진행 규칙입니다. 로그 파일 자체는 편집하지 말고, 실제 AI 대화를 원본 그대로 저장해야 합니다.

## 시작 전 상태

- 기업은 삼일PwC 하나만 다룹니다.
- 문제는 내부통제 문서 검토의 분산·수작업 병목입니다.
- 플러그인 목적은 누락 증빙 탐지, 불일치 경고, 검토 구조화입니다.
- 비밀정보나 실제 고객자료는 입력하지 않습니다.

## 시작 문장

```text
이번에 삼일PwC의 내부에서 발견된 문제 상황을 AX로 플러그인을 만들어 개선하려 하거든. 내부통제 문서 검토할 때 계약서, 전표, 승인 기록이 여기저기 다 찢어져 있어서 누락된 증빙이나 불일치하는 걸 사람이 눈으로 일일이 찾느라 시간 다 버리는 걸로 문제 상황은 정리됐어.
이제 이걸 그냥 문서로만 끝내지 말고 진짜 Codex 플러그인으로 구현해 봐야 하잖아. 그러려면 우선 보고서 내용 기준으로 문제를 훨씬 구체적으로 쪼개야 하고, 정확히 어떤 사용자가 어느 타이밍에 턱 막히는지부터 좁혀나가야 할 것 같아
```

## 보고서 넣는 문장

```text
내가 밑에 보고서 올려줄 테니까, 이거 참고해서 플러그인 스펙으로 뺄 수 있게 분석 빡세게 한 번 해줘. 분석이 끝나면 거기서 멈춰줘. 이후 질문은 내가 순서대로 하나씩 직접 던질게.
```

## 질문 진행 순서

아래 질문은 사용자가 직접 순서대로 입력합니다. Codex가 먼저 다음 질문을 제시하지 않습니다.

### 1. 문제 정의: 우리가 해결하려는 진짜 본질

- 이 보고서에서 지적하는 가장 결정적인 핵심 문제는 뭐라고 봐?
- 이걸 단순한 문서 과잉이 아니라, 실제 업무 관점에서 다시 날카롭게 정의하면 어떻게 표현할 수 있을까?
- 이 비효율 때문에 현업에서 실제로 어떤 업무적 실패 즉 휴먼 에러, 감사 지적, 마감 지연 등가 발생하고 있어?
- 이 전체 문제 중에서 우리 플러그인이 가장 확실하게 해결할 수 있는 스윗 스팟은 어디일까?
- 이 문제를 해결했을 때, 실무자가 체감할 수 있는 진짜 효과와 가치는 무엇일까?

### 2. 사용자와 상황: 누가, 언제 막히는가

- 이 문제로 인해 가장 크게 고통받고 야근하는 사용자가 정확히 누구야?
- 그럼 우리가 타깃으로 삼을 1차 유저는 누구로 좁히는 게 맞을까
- 그 사용자가 어떤 상황, 어떤 타이밍에 가장 힘들어해?
- 업무를 진행하다가 정확히 어느 지점에서 턱 막히거나 뇌정지가 오게 될까?
- 그 사람이 이 문제를 기필코 해결하고 싶어 하는 결정적인 업무 동기가 뭘까?

### 3. 업무 흐름과 병목: 시간을 잡아먹는 구간

- 1차 타깃 사용자가 내부통제 검토 업무를 할 때, 처음부터 끝까지 거치는 단계별 프로세스를 그려보자
- 그 흐름 중에서 가장 반복적이고 비효율이 극심한 병목 구간은 어디야?
- 지금은 툴이 없어서 이 병목을 어떤 식으로 꾸역꾸역 처리하고 있어?
- 그런 기존 방식의 한계는 뭐고, 왜 결국 사람이 직접 확인할 수밖에 없었던 걸까?
- AI 보조 기능을 도입한다면, 가장 먼저 적용해서 확실하게 효과를 볼 수 있는 단계는 어디일까?

### 4. 플러그인 기능 설계: 어떻게 도와줄 것인가

- 이 플러그인의 정체성은 단순 요약이 아니라, '누락 증빙 탐지 및 불일치 경고'로 포커스를 맞추는 게 맞겠지?
- AI가 기계적으로 잘 찾아낼 수 있는 영역과, 사람이 전문성을 가지고 최종 판단해야 하는 영역을 어떻게 나눌까?
- AI는 어떤 형태로 위험을 경고해 줘야 하고, 사람은 그걸 보고 어떤 의사결정을 내려야 할까?
- 현실적인 구현을 위해, 입력 자료를 계약서, 전표, 승인 기록, 엑셀 정도로 제한해도 실무에서 유용할까?
- 출력이 나올 때, 체크리스트, 위험 플래그, AI의 추가 질문 중 어떤 형식이 가장 실무적이고 직관적일까?

### 5. 구현 범위와 안전성: 리스크 관리와 스펙 제한

- 정보가 부족하거나 형식이 꼬였을 때, AI가 억지로 추측하지 않고 어떻게 멈추거나 사용자에게 되묻게 할까?
- AI가 자의적으로 과도하게 판단하지 않도록 제한해야 하는 핵심 규칙은 무엇일까?
- 혹시 모를 에러가 나왔을 때 유저가 신뢰할 수 있도록, 근거와 출처를 어떤 방식으로 보여주는 게 좋을까?
- 당장 구현 가능한 MVP와, 나중에 확장할 고도화 범위를 현실적으로 나눠보자

### 6. 검증과 완성도: 진짜 해결된다는 증명

- 플러그인을 테스트할 정상 케이스, 누락 케이스, 모순(불일치) 케이스를 하나씩 구체적으로 세팅해 볼까?
- 각 케이스를 넣었을 때, 플러그인이 각각 어떤 아웃풋을 내야 성공이라고 볼 수 있을까?
- 심사위원이나 유저에게 검증이 끝났다고 설득하려면, 실제 어떤 지표를 가지고 테스트해야 할까?
- 이 검증 결과를 어떤 논리 구조와 문장으로 작성해야 전문성이 살까?
- 이 플러그인이 문제를 해결할 수 있다는 걸 정량적으로 보여주기 위해 어떤 실험 결과, 예를들어 검토 시간 80% 단축 등을 제시해야 할까?
- 최종적으로 우리가 만든 이 플러그인을 단 한 문장으로 명쾌하게 정의한다면 뭐라고 할래?

## Codex 응답 원칙

- 공개 자료에서 확인 가능한 내용과 추론을 구분합니다.
- 사용자의 말투에 맞춰 자연스럽고 실무 기획 대화처럼 답합니다.
- "문서가 많다"가 아니라 어떤 업무 실패가 생기는지 날카롭게 설명합니다.
- 보고서 분석 응답이 끝난 뒤 다음 질문을 먼저 던지지 않습니다.
- 각 질문에 답한 뒤에도 다음 질문을 유도하지 않고, 사용자의 다음 입력을 기다립니다.
- 1차 타깃은 내부감사 담당자 또는 감사 보조 인력으로 유지합니다.
- 플러그인은 최종 감사 판단을 하지 않고, 사람의 검토를 돕는 구조로 설명합니다.
- 구현 범위는 최소 구현과 고도화 범위로 나눕니다.
- 검증은 정상 케이스, 누락 케이스, 모순 케이스로 설명합니다.
- 마지막 질문이 끝나면 제출 로그 종료 지점을 안내합니다.

## 마지막 응답 종료 문장

```text
여기까지가 제출 로그에 포함할 대화 흐름의 종료 지점입니다. 이 이후의 작업 지시나 패키징 확인은 별도 로그로 분리하는 것이 안전합니다.
```
